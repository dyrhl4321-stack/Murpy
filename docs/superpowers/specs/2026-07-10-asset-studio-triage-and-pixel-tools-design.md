# 에셋 스튜디오: 선별 화면 + 픽셀 수작업 도구 (설계)

2026-07-10 · 대상: `tools/asset-studio`

## 문제

아이템 하나에 사람 손이 21분 든다. 도구가 스스로 세운 1순위 지표는 60초다.

`top_rugby` 마지막 세션(`review_queue/top_rugby_20260710-010125/`) 실측:

| 항목 | 값 |
|---|---|
| 사람 시간 | 1,285초 (목표의 21배) |
| 편집 횟수 | 136회 |
| 브러시로 **추가**한 픽셀 | 37,619 |
| 브러시로 **지운** 픽셀 | 777 |
| 기계 검증 | **실패** (`no_body_overpaint`, `frame_area_stable`) |

같은 아이템의 직전 세션(439초, 43회 편집)은 검증을 통과했다.
**21분을 더 들여 자산 품질을 떨어뜨렸다.** 두 `preview.png` 는 육안으로 구분되지 않는다.

### 원인 1 — 수작업의 98%가 엔진이 버린 것을 되살리는 일

브러시 편집의 98%(37,619 / 38,396)가 "추가"다. 지우기가 아니다.
추가된 픽셀의 성분:

- **91.2%** 가 `|base − worn| ≤ 85` — 엔진이 원리적으로 볼 수 없는 픽셀
- **41.7%** 가 검은 테두리
- **49.7%** 가 살색

`engine/trimap.py:225` 의 규칙이 원인이다.

```python
fg = changed & alpha_ok & ~skin        # changed = |base - worn| > 85
```

픽셀아트의 검은 테두리는 base 에도 worn 에도 같은 검은색이다.
옷 경계에 붙은 테두리 **12,123px 의 100%** 를 자동추출이 버린다.
그 픽셀들의 diff 중앙값은 **26** — 임계값 85 의 3분의 1이다.

사람이 그걸 손으로 되살리면, 그중 16,683px 이 `still_body` 로 분류되어
검증기의 `no_body_overpaint` 게이트를 깨뜨린다.
**도구가 사람의 올바른 수정을 위반으로 판정한다.** 이것이 "지우면 뜯기고, 메꾸면 몸을 덮는" 무한루프다.

### 원인 2 — 축소 사고가 마스크 품질 판정을 오염시킨다

배포본 `char/items/*.png` 14개 중 기계 검증 통과는 `acc_airpodsmax` 하나뿐이다.
그런데 실패 사유 대부분이 마스크가 아니라 **LANCZOS 축소 잡음**이다.

알파를 `>128` 로 굳히고 조각 수를 다시 세면 판정이 뒤집힌다.

| 아이템 | `alpha>40` 조각 | `alpha>128` 조각 | 실제 |
|---|---|---|---|
| `bottom_bermuda` | 23 | **11** | 정상 (halo 먼지였음) |
| `top_redhood` | 46 | **14** | 손봐야 함 |
| `shoes_white` | 11 | **6** | 정상 |
| `hat_beanie` | 23 | **14** | 손봐야 함 |

**멀쩡한 아이템을 "부스러졌다"고 오진하고 있었다.** 사람이 볼 목록 자체가 틀렸다.

### 원인 3 — 1블록 미만의 편집은 앱에 도달하지 못한다

작업 시트는 846×1792. 배포본은 **423×896** (정확히 절반).
**시트의 2×2 블록 = 배포본의 1픽셀.**

따라서 1px 브러시 획은 축소되며 사라지거나 반투명 잡티로 남는다.
원형 브러시는 홀수 경계를 만들어 경계마다 반투명을 뿌린다.
136회 편집 중 상당수가 애초에 화면에 도달할 수 없었다.

## 목표

1. 사람이 **어떤 아이템의 어떤 프레임**을 봐야 하는지 기계가 짚어준다.
2. 사람이 손대야 할 때, **픽셀아트에 맞는 도구**로 빠르게 끝낸다.

비목표: 완전 자동화. 추출 엔진 재설계. AI 분할.

## 결정

**진단은 항상 `alpha>128` 로 굳힌 사본에서 한다.**
축소 잡음이 마스크 품질 판정을 오염시키지 못하게 막는 단일 규칙이다.

**테두리 흡수 패스는 채택하지 않는다.**
"옷 FG 에 인접한 어두운 픽셀을 반복 흡수" 를 프로토타입으로 실측했다.
사람 최종본과의 IoU 가 0.8171 → 0.8158 로 **떨어졌다**.
흡수한 5,725px 중 4,078px 이 과잉이었고, 사람이 칠한 35,195px 은 여전히 비었다.
되살릴 픽셀의 절반이 살색이라 어두운 픽셀 흡수로는 닿지 않는다. 근거 부족으로 기각.

## 구성 요소

### 1. `engine/triage.py` — 진단서 생성

```python
diagnose(item_path: Path, source: Path | None) -> Diagnosis
```

`alpha>128` 이진화 후 프레임 12칸 각각의 조각 수·면적을 잰다. 상태를 넷으로 분류한다.

| 상태 | 조건 | 누가 고치나 |
|---|---|---|
| `OK` | 모든 게이트 통과 | — |
| `MACHINE_FIXABLE` | 반투명·halo 만 문제, 굳힌 마스크는 정상 | 기계 (NEAREST 재축소) |
| `NEEDS_HUMAN` | 굳혀도 조각 > 12 또는 면적 CV > 0.15 | **사람** |
| `NEEDS_REEXTRACT` | 빈 프레임 있음, 또는 846 원본 없음 | 재추출 |

반환에 문제 프레임 목록을 담는다: `[{row, col, name: "아래·걸음B", components, area, flags}]`

`validate.py` 는 시트 전체 통과/실패만 낸다. 프레임 단위 지목은 여기서 새로 한다.

### 2. `server.py` — `GET /api/triage`

`char/items/*.png` 를 스캔한다. `*_thumb.png` 는 제외한다.
각 아이템의 846×1792 원본은 `review_queue/*/metadata.json` 의 `itemId` 로 역추적한다.
원본이 없으면 재축소가 불가능하므로 그 사실을 함께 보고한다.

파일 mtime 으로 결과를 캐시한다.

### 3. 선별 화면 — 새 첫 화면

한 줄 = 한 아이템. 썸네일, 상태 배지, 문제 프레임 칩.

**칩을 누르면 기존 편집 화면이 그 프레임에 바로 열린다.**
편집 화면은 `?item=<id>&frame=r0c2` 진입점만 새로 받는다. 나머지 로직은 그대로.

지금은 매번 12프레임을 처음부터 훑는다. `top_redhood` 처럼 특정 프레임만 부스러진 경우가 대부분이다.

### 4. 픽셀 수작업 도구 3종

기존: 원형 브러시 `B`, 지우개 `E`, 연결성분 삭제 `C`.

**(a) 격자 스냅 사각 브러시** — 원형을 대체한다.
크기를 픽셀이 아니라 **블록 단위**(1블록 = 2×2)로 센다. 언제나 짝수 경계에 스냅한다.
축소 후 사라지거나 반투명을 남기는 획을 원천적으로 막는다.

**(b) 연결영역 집기 (허용오차 마술봉)** — 가장 큰 시간 절감.
worn 이미지에서 클릭 지점과 색이 비슷하게 이어진 영역을 통째로 마스크에 넣거나 뺀다.
검은 테두리 12,123px 을 클릭 몇 번에 잡는다.

**허용오차는 필수다.** 상의 영역 worn 의 고유색이 **90,420개**다.
인접 픽셀끼리도 미세하게 다르므로 완전 일치 플러드필은 작동하지 않는다.
기본 허용오차는 실측으로 정한다(초기값 24, `|Δrgb|` 합).

**(c) 사각 선택 + 채우기/지우기** — 살색 덩어리(되살린 픽셀의 49.7%)를 쓸어담는다.
선택 영역도 2×2 격자에 스냅한다.

직선 도구는 이번 범위에서 뺀다.

### 5. 불변 규칙 (유지)

- base 시트는 읽기 전용.
- 아이템 RGB 를 만들어내지 않는다. 편집 상태는 마스크뿐, 픽셀은 worn 에서 복사.
- 브러시·마술봉·채우기 모두 **worn 이 불투명한 곳에서만** 작동한다.
- 모든 리샘플링은 정수배 NEAREST.
