# 머피룸 재정비 — 배경 · 가구 스케일 · 충돌 시스템 설계

> 2026-07-22 브레인스토밍 확정. 대표 승인: 벽 3타일, 크림 벽지, Pixel Interiors로 가구 재구축.

## 배경 — 왜 하는가

대표 지적: 기존 LimeZu 가구가 캐릭터 대비 너무 작다(특히 거울·전등·소파). 실측으로 확인됨.

캐릭터를 170cm로 보면 1유닛 ≈ 1.08cm. 이 기준 환산 결과:

| 아이템 | 현재 | 환산 | 실제 가구 | 필요 배율 |
|---|---|---|---|---|
| mirror_stand | 60×96 | 65×103cm | 50×170cm | **1.65×** |
| shelf_books | 96×96 | 103×103cm | 80×180cm | **1.74×** |
| wardrobe | 81×117 | 87×126cm | 100×200cm | **1.59×** |
| lamp_blue | 45×99 | 48×107cm | 40×160cm | **1.50×** |
| plant_palm | 69×93 | 74×100cm | 60×150cm | 1.50× |
| sofa_brown | 135×64 | 145×69cm | 200×85cm | 1.23× |
| globe | 84×63 | 90×68cm | 30×40cm | **0.59×** (과대) |
| table_wood | 114×111 | 123×119cm | 120×75cm | 0.63× (과대) |

**근본 원인:** 캐릭터가 3.3타일(158유닛)로 유난히 크다. 일반적인 타일셋 아트는 캐릭터 1.5~2타일을 전제하므로, 어떤 에셋 팩을 가져와도 같은 문제가 반복된다. 즉 **스케일 보정은 에셋 소스와 무관하게 필요한 작업**이다.

**캐릭터를 줄이지 않는 이유(대표 결정):** `_charApplyPos`는 헬스장·테니스·골프 필드와 스쿼드 워킹룸에서도 쓰인다. 캐릭터를 줄이면 그 배경들이 전부 어색해진다. 영향 범위가 방 안으로 한정되는 가구 쪽을 조정한다.

## 에셋 소스 결정

대표 보유 팩 중 사용 가능한 것을 실사한 결과:

- **Pixel Interiors 32x32pxl.png** (544×352) — 침대·옷장·책장·안락의자·테이블·의자·책상·러그·문·주방·냉장고·거울·화분·라디에이터. 스케일 일관성 있고 종류 풍부 → **가구 소스로 채택**
- **sierrassets floors and walls** (135×155, 8×8타일/9px피치) — 실사 결과 **탑다운 방 템플릿**(바깥 테두리=벽, 안쪽=바닥, 아래 홈=출입구). 우리 방은 벽면을 정면으로 보는 사선 시점이라 원근이 맞지 않아 **미채택**
- Pixel Interiors의 벽 영역도 동일하게 탑다운 프레임 형태 → 미채택
- 나머지(plants, pets, TILESET INDOORS +Bathroom)는 이번 범위 밖. pets의 애니메이션 고양이·토끼는 향후 펫 기능 후보로 기록만 해둔다

**결론:** 두 팩 모두 정면 벽면 에셋이 없다. 현재 `field_workshop.png`의 어색한 상단 띠(LimeZu 침대 스프라이트를 벽처럼 반복한 것)도 같은 한계에서 나온 것으로 보인다. **바닥은 팩에서 쓰고 벽만 코드로 생성**한다.

**라이선스:** 두 팩 모두 상업적 게임 사용 허용, 에셋 자체의 재판매·재배포 금지. 다만 Murpy는 공개 GitHub 저장소로 배포되어 원본 PNG가 직접 다운로드 가능한 상태다. "게임에 사용"과 "파일 재배포"의 경계에 걸치는 지점으로, 투자 실사 전 정리 필요. 이 스펙의 범위 밖이며 `docs/사업계획/legal/`에서 다룬다.

## Phase 1 — 방 배경 재제작

**산출물:** `char/fields/field_room.png` (768×768) + 생성 스크립트 `char/build_room_field.py`

- **벽 3타일(144유닛)**: 크림 벽지(#DED2BA 계열) + 세로 스트라이프 + 상단 몰딩 + 나무 걸레받이(22유닛)

- 벽지 색은 스크립트 파라미터로 분리 — 나중에 "벽지 바꾸기" 기능으로 확장 가능한 구조
- `_FIELDS.home.map` 갱신: 막힌 행을 0~1행 → **0~2행**으로

**교체할 참조 3곳** (전부 `field_workshop.png` 하드코딩):

## Phase 2 — 가구 카탈로그 재구축

**산출물:** `char/rooms/pi_*.png` + 슬라이스 스크립트 `char/slice_pi_furniture.py`

- Pixel Interiors 스프라이트를 **4배(정수배)** 로 추출. 근거: 문 스프라이트 23×42px를 실제 200cm로 보면 캐릭터 기준 배율 4.4배가 나오나, 정수배 유지를 위해 4배 채택. 결과 옷장·책장 96×184로 캐릭터(158)보다 커져 비율이 성립
- 카탈로그 ~20종: 침대·옷장·책장·소파·안락의자(색상별)·테이블·의자·책상·러그·화분·거울·TV·냉장고·주방
- **기존 LimeZu 15종은 교체.** 저장된 방이 대표 계정 하나뿐이라 마이그레이션 불필요(대표 확인)
- `ROOM_ITEMS` 전면 교체
- **한정 오브젝트 5종**(`s2608_*`, `season_box`)도 새 스케일 기준으로 w/h 재산출. 현재 값(덤벨 158×98 등)은 구 스케일 기준이라 무효

한정 오브젝트는 나노바나나 생성물이라 Pixel Interiors와 화풍이 다르다. 이는 의도된 것으로, 기믹 아이템이 눈에 띄는 편이 낫다는 대표 판단에 따른다.

## Phase 3 — 충돌 시스템

현재 `_charWalkable`은 정적 맵(`f.map`)만 읽고, 배치된 가구는 충돌 계산에 전혀 반영되지 않는다. 이를 **정적 맵 + 동적 가구 오버레이**로 확장한다.

**아이템별 `block` 속성 (기본 자동 + 예외 명시):**

- 미지정 시 기본값 = 스프라이트 **하단 30% 밴드**를 바닥 면적으로 간주해 해당 타일 블록
- 예외만 개별 지정:
  - 러그류 → `none` (밟고 지나감)
  - 벽걸이 TV·액자·세계지도 → `none` (벽에 붙어 있음)
  - 거울 → 얕게 (앞에 바짝 서서 비춰보는 느낌. 하단 10~15%)
  - 침대·옷장 → 전체 하단 밴드 유지
- 매달 한정 아이템이 추가되는 운영 구조이므로, **아무것도 안 적어도 그럴듯하게 동작하는 기본값**이 핵심

**오버레이 계산 시점:** 저장이 아니라 **렌더 시점**에 계산한다. 저장 데이터(`{id,x,y}`)는 그대로 두고 규칙만 바꿀 수 있어야, 나중에 블록 값을 조정해도 기존 방이 깨지지 않는다.

**갇힘 해소:** 캐릭터가 막힌 칸 위에 있게 되면(자기 위에 가구를 배치하는 경우 등) 가장 가까운 빈 칸으로 밀어낸다.

## 범위 제외

- 캐릭터 크기 변경 (다른 필드 파급 때문에 제외)
- 벽지 변경 UI (이번엔 기본값 하나만. 구조만 확장 가능하게)
- 펫 기능 (sierrassets 애니메이션 고양이·토끼는 후보로만 기록)
- 다른 필드(헬스장·테니스·골프)의 가구·충돌
- 에셋 라이선스 대응 (사업계획 문서에서 별도 처리)

## 검증 방법

이 프로젝트에는 테스트 하네스가 없다. 각 Phase는 다음으로 검증한다:

- Phase 1·2: 합성 미리보기 PNG를 생성해 대표 승인 후 등록 (기존 관행)
- Phase 3: 배포 후 실앱에서 직접 걸어보며 확인 — 가구 통과 여부, 러그 통과, 거울 접근, 갇힘 해소
