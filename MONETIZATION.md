## 0. 현황 요약 (코드 기반)

코드를 읽고 발견한 사실: **수익화 UI는 이미 80% 완성되어 있다.**

| 기능 | UI 상태 | 실제 결제 |
|------|---------|-----------|
| PRO 구독 (월 9,900원) | ✅ 완성 | ❌ 없음 (`isPro=true` 만 설정) |
| 크루 부스트 (5,900~29,900원) | ✅ 완성 | ❌ 없음 (토스트만 표시) |
| 크루 입장료 (크루장 80% / Murpy 20%) | ✅ 완성 | ❌ 없음 (토스트만 표시) |
| 정모 선입금 | ✅ UI 있음 | ❌ 없음 |
| 광고 슬롯 (`id="ad-slot"`) | ✅ `display:none` 대기 중 | ❌ 연동 없음 |
| 수익 트래킹 (`earnings` 필드) | ✅ 데이터 구조 있음 | ❌ 미집계 |

결론: 지금 당장 필요한 것은 새 기능 개발이 아니라 **결제 레이어 1개를 끼워 넣는 것**이다.

## 1. 현재 코드 기반 수익화 가능 지점

### 1-1. 크루 (★★★★★ 최고 잠재력)

코드에 이미 구현된 수익 구조:
```
입장료 설정: [0, 1,000, 2,000, 3,000, 5,000, 9,900원]
분배: 크루장 80% / MURPY 20%  ← 이미 코드에 명시됨
```

**수익 경로:**
- 크루 입장료: 유저가 가입할 때 1회 결제 → Murpy 20% 수취
- 크루 부스트: 크루장이 탐색 상단 노출을 구매 (5,900 / 9,900 / 29,900원)
- 정모 선입금: 참가자가 모임 참가비를 앱 내 결제 → 불참 시 공금 처리

**현실적 계산 (GBD CREW 247명 기준):**
- 입장료 2,000원 × 신규 가입자 월 20명 = 40,000원 → Murpy 8,000원
- 부스트 9,900원 × 크루 10개 = 99,000원
- 정모 선입금 5,000원 × 정모 4회 × 참가자 30명 = 600,000원 → Murpy 20% = 120,000원

작아 보이지만 이게 **유저 300명 이전에 검증할 핵심 지표**다.

---

### 1-2. PRO 구독 (★★★★)

이미 UI 완성. 기능 구성도 합리적:

| 제한 기능 | 무료 | PRO |
|----------|------|-----|
| 좋아요 | 하루 5번 | 무제한 |
| 댓글 | 3개 | 무제한 |
| 매칭 후 먼저 말 걸기 | ❌ | ✅ |
| 대숲 신청자 보기 | ❌ | ✅ |
| 나한테 좋아요 보기 | 블러 | 블러 해제 |

**유일한 문제:** `activatePro()` 함수 내부가 `isPro = true;` 한 줄이다.  
결제 레이어만 붙이면 즉시 수익화 가능.

**월 9,900원 vs 시장 비교:**
- 틴더 플러스: 12,000원
- 소모임 PRO: 없음 (광고 기반)
- Murpy PRO: 9,900원 → **합리적 포지션**

---

### 1-3. 매칭 (★★★)

현재 더미데이터만 있음. Firestore 연동 후 수익화 포인트:
- 슈퍼라이크 (1일 1회 무료 → 추가 구매)
- 거리 필터 조절 (PRO 기능)
- 이미 본 사람 다시 보기 (PRO 기능)
- "나한테 좋아요" 블러 해제 → PRO 가입 유도 (이미 UI 있음)

---

### 1-4. 센터 (★★)

현재 더미데이터. 연동 후:
- 센터 프리미엄 등록 (상단 노출, 월정액)
- 센터 광고 배너 (CPA/CPM)
- PT 예약 수수료 (장기 목표)

---

### 1-5. 홈 피드 광고 (★★)

`id="ad-slot"` 이 이미 `display:none`으로 준비됨.  
유저 1,000명+ 되면 Google AdSense 또는 운동 브랜드 스폰서드 포스트 삽입 가능.

## 2. 클래스/레슨 마켓플레이스 모델 분석

### 2-1. 비즈니스 모델 구조

```
트레이너가 클래스 개설
    ↓
수강생이 앱 내 결제
    ↓
Murpy 수수료 공제 (예: 15~20%)
    ↓
트레이너 정산 (D+7 또는 D+15)
```

이 모델은 **기술적으로 구현 가능하나, 비기술적 장벽이 훨씬 크다.**

---

### 2-2. 결제 시스템 (한국 기준)

**추천: 토스페이먼츠**
```
이유:
- 개인사업자 없이도 테스트 환경 사용 가능
- SDK가 가장 깔끔 (React 없이도 vanilla JS 지원)
- 카카오페이, 토스, 신용카드 모두 단일 SDK로 처리
- 정산 API 제공 (트레이너 계좌로 자동 송금 가능)
- 월 거래액 1억 미만 무료 요금제 있음
```

**2순위: 카카오페이**
```
이유:
- 이미 카카오 로그인 연동됨 → 결제 동선 자연스러움
- 단점: 단독 결제 모듈로는 카드/다른 간편결제 미지원
```

**구현 최소 코드 예시 (토스페이먼츠):**
```javascript
// 백엔드 없이도 프론트에서 결제 요청 가능 (단, 검증은 반드시 서버에서)
const tossPayments = TossPayments('your_client_key');
tossPayments.requestPayment('카드', {
  amount: 9900,
  orderId: 'order-' + Date.now(),
  orderName: 'Murpy PRO 1개월',
  successUrl: location.href + '?success=true',
  failUrl: location.href + '?fail=true',
});
```

---

### 2-3. 정산/세금/사업자 관련 현실적 고려사항

**결제 수취를 위해 필요한 것:**

| 항목 | 내용 |
|------|------|
| 사업자 등록 | 필수. 개인사업자로 시작 (자유업종 "앱 서비스 운영") |
| PG사 계약 | 토스페이먼츠 심사 약 3~5 영업일 |
| 통신판매업 신고 | 구청 온라인 신청, 2~3일 소요 |
| 에스크로 의무 | 5만원 이상 결제 시 (클래스 결제에 해당 가능) |

**트레이너 정산 시 추가 이슈:**
- 트레이너가 개인사업자가 아니면 **원천징수 3.3%** 공제 후 지급 (사업소득)
- 분기별 지급명세서 제출 의무
- 연 매출 2,400만원 초과 시 부가세 신고 의무
- → **단기에는 트레이너 정산 기능 피하는 게 현실적**

**더 현실적인 클래스 수익 모델:**
```
트레이너가 크루 개설 + 입장료 설정
→ 기존 크루 입장료 구조 그대로 활용
→ 추가 개발 없이 "클래스" 개념으로 마케팅
```

## 3. 수익 모델 우선순위 제안

### Phase 1: 단기 (지금 ~ 유저 300명) — "결제 레이어 삽입"

**목표: 월 50~100만원 ARR 검증**

1. **PRO 구독 결제 연동** (토스페이먼츠, 2~3일 작업)
   - `activatePro()` 함수에 결제 API 콜 추가
   - Firebase `users/{uid}` 에 `isPro`, `proExpireAt` 필드 추가
   - 월 9,900원 → 월 100명이면 월 990,000원

2. **크루 부스트 결제 연동** (1~2일 작업)
   - `openBoostModal()` → `boostCrew()` 에 결제 API 콜 추가
   - 가장 간단한 일회성 결제 (정산 불필요)

3. **크루 입장료 흐름 정리** (1주일 작업)
   - 현재 토스트만 표시 → 실제 결제 + Firestore 기록
   - 크루장 수익 대시보드 (이미 `earnings` 필드 있음)
   - 단, 크루장 정산은 일단 "플랫폼 포인트" 또는 "수동 송금"으로 시작

**이 단계에서 백엔드 서버 없이 가능한 이유:**
- 결제 요청/완료 확인은 프론트에서 가능
- Firestore로 상태 저장
- 단, 결제 검증(위변조 방지)은 Firebase Functions로 처리 (서버리스)

## 4. 기술적 로드맵

### 4-1. 결제 연동에 필요한 것 (현재 → PRO 결제 가능)

```
현재 상태:
  index.html (단일 파일)
  Firebase Firestore (DB)
  Firebase Auth (로그인)

추가 필요:
  1. 토스페이먼츠 클라이언트 키
  2. Firebase Functions 1개 (결제 검증용)
  3. Firestore users 컬렉션에 isPro, proExpireAt 필드
  4. 사업자등록 + PG사 계약
```

**최소 구현 (Firebase Functions 사용):**
```javascript
// Firebase Function (Node.js)
exports.verifyPayment = functions.https.onCall(async (data, context) => {
  const { paymentKey, orderId, amount } = data;
  // 토스페이먼츠 서버 검증
  const response = await fetch('https://api.tosspayments.com/v1/payments/confirm', {
    method: 'POST',
    headers: { Authorization: 'Basic ' + btoa(secretKey + ':') },
    body: JSON.stringify({ paymentKey, orderId, amount })
  });
  if (response.ok) {
    // Firestore 업데이트
    await db.collection('users').doc(context.auth.uid).update({
      isPro: true,
      proExpireAt: new Date(Date.now() + 30 * 24 * 60 * 60 * 1000)
    });
    return { success: true };
  }
  throw new functions.https.HttpsError('failed-precondition', '결제 실패');
});
```

---

### 4-2. 단일 HTML + Firebase 구조의 한계

| 한계 | 영향 | 해결책 |
|------|------|--------|
| 결제 시크릿 키 노출 불가 | 서버 없으면 결제 검증 불가 | Firebase Functions (서버리스) |
| 정산 자동화 | 크루장 계좌 송금 불가 | 토스페이먼츠 정산 API + Functions |
| 웹훅 수신 불가 | 정기 결제 갱신 확인 불가 | Functions HTTP 엔드포인트 |
| 코드 보안 | API 키 전체 노출됨 | Functions로 민감 API 이동 |
| 파일 크기 (272KB+) | 모바일 초기 로딩 느림 | 단기 무시, 유저 1,000명 후 분리 |

**핵심:** 단일 HTML로도 결제 MVP는 가능하다. Firebase Functions만 추가하면 된다.  
풀 백엔드(Express, Next.js 등)는 크루 정산 자동화, 정기 구독 관리, 세금계산서 발행이 필요해지는 **유저 500명+ 단계**에서 고려하면 충분하다.

---

### 4-3. 결제/정산이 들어가면 백엔드 서버가 필요한 이유

```
프론트엔드에서 처리 가능:
  ✅ 결제 요청 (리다이렉트 방식)
  ✅ 결제 완료 화면 표시
  ✅ 간단한 상태 저장

백엔드 없으면 절대 불가:
  ❌ 결제 검증 (paymentKey 서버 확인) → 위변조 방지 필수
  ❌ 시크릿 키 보관 (클라이언트 노출 시 악용 가능)
  ❌ 정기 결제 갱신 웹훅 수신
  ❌ 크루장 정산 자동화 (은행 API 연동)
  ❌ 세금계산서 자동 발행
  ❌ 환불 처리 자동화
```
